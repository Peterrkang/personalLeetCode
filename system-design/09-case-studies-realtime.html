Week 9 — Case Studies: Chat & Notifications

Real-time systems: persistent connections, delivery guarantees, ordering. The deep-dive interviewers pick here is almost always "how do you guarantee the message arrives, exactly once, in order?"

Case 1 — Chat system (WhatsApp / Messenger)

Requirements

  • Functional: 1:1 chat, group chat, delivery/read receipts, online presence, offline delivery (messages arrive when you reconnect), chat history.
  • Non-functional: delivery latency < ~100 ms; no message loss ever (this is the consistency-critical one — contrast with feeds); ordering within a conversation.

Connections & routing

  • Clients hold a WebSocket to a chat server. These servers are stateful (they own live connections) — the interesting scaling problem.
  • 1M connections/server is achievable; a service-discovery / session registry (Redis or ZooKeeper) maps user_id → chat server.
  • Message path: A sends to server-1 → server-1 looks up B's server → forwards to server-2 → pushes down B's socket. If B is offline: persist to B's inbox queue; deliver on reconnect (+ push notification via case 2).

Delivery guarantees (the deep dive)

  • ACK at every hop: client→server ack (sender shows ✓), server persists before acking (never lose an acked message), recipient-device ack (✓✓), read receipt (blue).
  • Sender retries until acked → duplicates possible → messages carry a client-generated ID; receivers dedupe. At-least-once + idempotency again.
  • Ordering: per-conversation sequence numbers assigned by the server (or one Kafka partition per conversation). Global ordering is neither possible nor needed — per-conversation is the contract.
  • Storage: write-heavy, append-only, queried by (conversation_id, seq) → wide-column store (Cassandra/HBase) partitioned by conversation_id. WhatsApp famously deletes after delivery; Messenger keeps history — say which you're building.

Presence & group chat

  • Presence: heartbeat every ~10 s → presence service; missing heartbeats → offline. Don't broadcast every flap: friends fetch presence on chat-open, subscribe only to open conversations.
  • Group chat: sender → server fans out to each member's server/inbox (small groups: fine; the fan-out is bounded by group size cap — this is why groups have member limits).

Case 2 — Notification system

  • Inputs: events from services ("order shipped", "X liked your post") → notification service → channels: mobile push (APNs/iOS, FCM/Android), email, SMS, in-app.
  • Architecture: producers → queue (per-channel topics) → channel workers → third-party gateways. Workers are stateless; queue absorbs spikes (breaking-news moment).
  • Third parties fail: retries + backoff, DLQ, and idempotency keys so retries don't double-send. Respect provider rate limits.
  • Dedup & preferences: check user settings + rate-cap per user ("max N/day") + collapse similar events ("A and 12 others liked…") before sending. A notification system's hardest job is not sending.
  • Track delivery status per notification for analytics and retry decisions.
Exercise (45 min): Design WhatsApp. Deep-dive: trace one message A→B through every ack, then kill B's chat server mid-delivery and show why the message still arrives exactly once. Follow-up: how does "last seen 5 min ago" work without hammering the presence service?

Self-check

  • Why WebSockets here, and what does statefulness break that stateless designs enjoy?
  • How does server-1 find which server holds B's connection?
  • Walk the three ack levels: sent / delivered / read.
  • How are duplicate messages prevented end-to-end?
  • Why is per-conversation ordering the right contract (not global)?
  • Why does a notification system need per-user rate caps and collapsing?
