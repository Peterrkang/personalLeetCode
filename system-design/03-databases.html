Week 3 — Databases

The database choice is the single most consequential decision in most designs. By end of week: pick SQL vs NoSQL and justify it, explain what an index actually does, and know ACID vs BASE beyond the acronyms.

1. SQL vs NoSQL

Relational (Postgres, MySQL)NoSQL
SchemaFixed, enforced up frontFlexible / evolving
QueriesRich: joins, aggregates, ad-hoc SQLSimple, by known access pattern (usually key lookups)
TransactionsStrong ACIDOften per-item only; BASE
ScalingVertical + read replicas; sharding is manual and painfulHorizontal sharding built in
Pick whenRelational data, complex queries, correctness first (orders, payments, inventory)Massive scale, high write throughput, simple access patterns (feeds, sessions, events)

NoSQL families: key-value (Redis, DynamoDB), document (MongoDB), wide-column (Cassandra — huge write throughput, queries by partition key), graph (Neo4j — relationship traversal).

  • Distributed SQL (Spanner, CockroachDB, TiDB, Vitess/PlanetScale) blurs the old table: full SQL + ACID with horizontal sharding built in. The catch: consensus on the commit path adds write latency, and cross-shard transactions still cost more. Modern senior answer to "relational but outgrows one machine" — name it instead of accepting "sharding SQL is painful" as a law of nature.
Interview default: start relational. Move a table to NoSQL when you can name the reason: write volume, size beyond one machine, or schema flexibility. "We'll use MongoDB because it's web scale" is an anti-signal; "user table stays in Postgres, activity events go to Cassandra because it's 50k writes/sec append-only" is the senior version.

2. Indexes

  • An index is a sorted copy of selected columns pointing back at the rows — usually a B-tree, so lookups and range scans are O(log n).
  • Without one: full table scan, O(n). With one: the difference between 5 ms and 30 s on a big table.
  • Cost: every index slows writes (each insert/update maintains every index) and eats disk. Index the columns you filter/join/sort by — not everything.
  • Composite index on (user_id, created_at) serves "this user's recent items" in one walk; order matters — it can't serve a query on created_at alone (leftmost-prefix rule).
  • Know the phrase covering index: if the index contains every column the query needs, the DB never touches the table.

3. ACID, transactions & isolation

  • Atomicity — all or nothing. Consistency — constraints hold before and after. Isolation — concurrent transactions don't see each other's partial work. Durability — committed means survives a crash (write-ahead log).
  • Isolation levels trade correctness for speed: read uncommitted → read committed → repeatable read → serializable. Most DBs default to read committed; know that lower levels admit anomalies (dirty reads, non-repeatable reads, phantoms).
  • BASE (the NoSQL counterpoint): Basically Available, Soft state, Eventually consistent — prefer answering every request over answering with the very latest value.

4. Normalization vs denormalization

  • Normalize (3NF): each fact stored once. No update anomalies; smaller data; but reads need joins.
  • Denormalize: copy data to where it's read (store author_name on the post row). Reads get fast and join-free; writes must update every copy and can drift.
  • Rule of thumb: normalize for writes/correctness, denormalize for reads/scale. At scale, read-heavy systems denormalize aggressively (a feed item embeds author name + avatar) and accept the update fan-out.

5. Choosing keys

  • Auto-increment IDs don't work across shards (collisions, coordination). Options: UUID (no coordination, but random → index-unfriendly), Snowflake-style IDs (timestamp + machine + sequence → unique, roughly sortable by time), or a central ticket server.
  • Time-ordered IDs are a superpower: "sort by ID" = "sort by creation time," which feeds and pagination want anyway.

Worked example — e-commerce orders schema

  • users(id, email, …), products(id, price, stock, …), orders(id, user_id, status, total, created_at), order_items(order_id, product_id, qty, unit_price).
  • Note unit_price is denormalized on purpose — the order must record the price paid, even if the product's price changes later.
  • Placing an order = one ACID transaction: insert order + items, decrement stock, fail atomically if stock insufficient → relational DB is the right call.
  • Indexes: orders(user_id, created_at) for "my orders"; order_items(order_id).
  • Product browsing (read-heavy, no transactions) can live elsewhere — search index, cache — while checkout stays in Postgres.
Exercise: You're storing 10B analytics events/month (append-only, queried by user_id + time range, no joins, no updates). Pick a database, the partition key, and explain why Postgres is the wrong default here — then explain what you lose by leaving it.

Self-check

  • What does a B-tree index make fast, and what does it cost?
  • Why can't (user_id, created_at) serve a query filtering only on created_at?
  • ACID vs BASE — what is each optimizing for?
  • When is denormalization the right call, and what's the price?
  • Why do sharded systems need Snowflake-style IDs?
