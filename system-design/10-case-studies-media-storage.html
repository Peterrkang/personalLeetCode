Week 10 — Case Studies: YouTube & Dropbox

Media and file systems: the data is huge, dumb, and immutable — so the design centers on pipelines, chunking, and CDNs rather than clever databases.

Case 1 — YouTube / Netflix

Requirements & estimates

  • Functional: upload video; watch (stream) video; metadata (titles, views, likes); search — mention, don't build.
  • Non-functional: smooth playback everywhere on any connection; upload can be slow but must be reliable. Reads ≫ writes (1000:1).
  • Estimate flavor: 500 hrs of video uploaded/min; a 10-min 1080p video ≈ 1 GB raw, a few hundred MB transcoded across renditions. Bandwidth, storage, and CDN egress dominate — this is a cost-engineering problem.

Upload & transcoding pipeline (the deep dive)

  1. Client gets a pre-signed URL and uploads directly to object storage in resumable chunks — app servers never touch the bytes.
  2. Upload-complete event → queue → transcoding pipeline (this is week 6's pipeline at industrial scale).
  3. Transcoder splits video into segments (~2–10 s each), transcodes each segment into a ladder of renditions (240p → 4K) in parallel across a worker fleet — a 2-hour movie transcodes in minutes because it's embarrassingly parallel. It's a DAG: split → transcode segments → also generate thumbnails/captions → stitch manifest → validate → publish.
  4. Output: segments + a manifest (HLS/DASH) listing every rendition's segment URLs → pushed to CDN.
  5. Video status: processing → ready; notify uploader async.

Playback — adaptive bitrate streaming

  • Player fetches the manifest, then requests segments over plain HTTP, choosing the rendition per segment based on measured bandwidth — degrade to 480p on bad Wi-Fi, back to 1080p when it recovers. No special protocol; it's all cacheable HTTP, which is why CDNs work.
  • Popular content sits in edge caches (Netflix ships appliances into ISPs); the long tail pulls from origin.
  • Metadata (title, counts, comments) is a normal read-heavy web service: DB + cache, view counts via buffered counters (week 8).

Case 2 — Dropbox / Google Drive (file sync & object storage)

The key idea: chunking

  • Files split into ~4 MB chunks, each identified by the hash of its content.
  • Delta sync: edit a 1 GB file → only changed chunks re-upload.
  • Dedup: chunk already known (same hash from any user)? Don't store it twice.
  • Resumable transfers: a dropped connection resumes at the next chunk.

Architecture

  • Block service: stores chunks in object storage, keyed by content hash. Dumb and infinitely scalable.
  • Metadata service: the brain — file → ordered chunk list, versions, folder tree, sharing ACLs. Relational DB, sharded by user/namespace. Consistency matters here (two devices editing metadata) even though chunk storage is trivially eventual.
  • Sync flow: client watches FS → computes changed chunks → uploads missing chunks → commits new metadata version atomically → notification service (long polling in Dropbox's design) tells other devices → they pull the delta.
  • Conflicts: two offline edits to the same file → don't merge, don't drop: keep both (report (Bob's conflicted copy)). Versioning = keeping old chunk lists — chunks are immutable, so history is nearly free.

Under the hood — distributed file systems (GFS / HDFS)

Dropbox's block service, S3, and Hadoop all descend from one design: Google File System (2003), cloned in open source as HDFS. Know its shape — it's the canonical answer to "how do you store petabytes on commodity machines?"

  • One master (HDFS: NameNode) keeps all metadata in memory — file → chunk list, chunk → which servers. Chunkservers (DataNodes) hold the actual bytes in large blocks (64–128 MB) — big on purpose, so a petabyte is few enough blocks for one master to track. Each block is replicated ×3 across racks.
  • Data bypasses the master: the client asks it for block locations (a tiny message), then streams directly from chunkservers. The master never touches file bytes — that's how one machine coordinates petabytes without being a bottleneck.
  • Failure handling: chunkservers send heartbeats; go silent long enough and the master re-replicates every block that server held from the surviving copies. Each block carries a checksum verified on read — a corrupt replica is discarded and re-copied from a good one.
  • Notice this is exactly the Dropbox split above — a small consistent brain over dumb wide storage — with week 12's heartbeats and checksums doing the real work.
Pattern to reuse everywhere: separate the data plane (huge, dumb, immutable, cache-friendly — chunks/segments on object storage + CDN) from the control plane (small, smart, consistent — metadata in a proper DB). YouTube, Dropbox, S3, GFS/HDFS, and Instagram all decompose this way.
Exercise (45 min): Design YouTube; deep-dive the transcoding DAG — how do you parallelize a single video, track per-segment failures, and retry without redoing the whole job? Then, for Dropbox: two devices edit the same file offline — walk both syncing back and exactly where the conflict is detected.

Self-check

  • Why do clients upload straight to object storage instead of through app servers?
  • Why segment videos before transcoding? (Two reasons: parallelism + ABR.)
  • How does adaptive bitrate streaming actually pick a quality?
  • What three features fall out of content-hash chunking almost for free?
  • Which part of Dropbox needs strong consistency, and which is happily eventual?
  • How do immutable chunks make file versioning cheap?
  • In GFS/HDFS, why do clients never stream file data through the master — and why are blocks so large?
