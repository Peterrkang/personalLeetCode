Week 1 — System Design Fundamentals

The vocabulary and mental models everything else builds on. By end of week: reason about scale, name the core tradeoff (consistency vs availability vs latency vs cost), and do back-of-envelope math without panicking.

1. Latency vs Throughput

  • Latency = time for one operation (e.g. 50 ms per request). What a single user feels.
  • Throughput = operations per unit time (e.g. 10,000 requests/sec). What the system sustains.
  • Related but distinct: you can lower latency AND raise throughput. Batching often increases throughput at the cost of latency.
  • Goal: minimize latency, maximize throughput — but they trade off, so optimize for the requirement.

2. Availability — "the nines"

Availability = % of time the system is up and serving correctly.

AvailabilityDowntime / yearDowntime / day
99% (two nines)3.65 days14.4 min
99.9% (three nines)8.77 hrs1.44 min
99.99% (four nines)52.6 min8.6 sec
99.999% (five nines)5.26 min0.86 sec
  • Achieved via redundancy (no single point of failure), replication, failover, and health checks.
  • Reliability (correct results / no failures) is related but not the same as availability (is it up).
  • SLA / SLO / SLI: SLA = the promise to customers (often with penalties); SLO = your internal target; SLI = the actual measured metric.

3. Scaling

  • Vertical (scale up): bigger machine. Simple, but a hard ceiling and a single point of failure.
  • Horizontal (scale out): more machines behind a load balancer. Near-unlimited, but adds coordination, partitioning, and consistency challenges.
  • Stateless services scale horizontally easily — any instance handles any request. Push state to a shared store. Stateful services are harder and need sticky sessions or careful partitioning.
  • Rule of thumb: keep app servers stateless; push state down to databases/caches/queues.

4. Latency numbers worth knowing (orders of magnitude)

OperationApprox time
L1 cache reference~1 ns
Main memory (RAM) reference~100 ns
Read 1 MB sequentially from memory~10 µs
SSD (NVMe) random read~20–100 µs
Read 1 MB from SSD (NVMe)~0.2–0.5 ms
Round trip within same datacenter~0.5 ms
Disk (HDD) seek — cold/archival tiers only now~10 ms
Round trip CA ↔ Netherlands~150 ms

Takeaways: memory is ~1,000× faster than even NVMe and ~100,000× faster than an HDD seek; network round trips are expensive — minimize them; stay in the same region; cache aggressively. (Assume NVMe SSDs everywhere in modern designs; spinning disks survive only in cold-storage tiers like S3 Glacier.)

5. Back-of-the-envelope estimation

A repeatable recipe:

  1. Users: DAU (daily active users).
  2. QPS: actions/user/day × DAU ÷ 86,400 sec. Then peak QPS ≈ 2–3× average.
  3. Storage: items/day × bytes/item × retention. Add a multiplier for replication (×3 common).
  4. Bandwidth: QPS × bytes/response.

Handy constants:

  • 1 day ≈ 86,400 sec (≈ 105).
  • Powers of 10: KB 103, MB 106, GB 109, TB 1012, PB 1015.
  • 1 char ≈ 1 byte (ASCII); a tweet ≈ 280 bytes; a small image ≈ 200 KB–1 MB.

Worked example — 100M DAU, each posts 2 items/day

  • Writes/day = 100M × 2 = 200M.
  • Avg write QPS = 200M ÷ 86,400 ≈ ~2,300 writes/sec. Peak ≈ ~5,000–7,000/sec.
  • If each item ≈ 1 KB: storage/day = 200M × 1 KB = 200 GB/day → ~73 TB/year (×3 replication ≈ ~220 TB/year).
  • If reads are 100× writes (typical social): read QPS ≈ 230k/sec → you'll need caching + read replicas.

The interview framework (memorize, use every time)

  1. Clarify functional + non-functional requirements (scale, latency, consistency, availability, read/write ratio).
  2. Estimate QPS, storage, bandwidth.
  3. API design — core endpoints.
  4. High-level design — clients → load balancer → services → DB / cache / queue.
  5. Deep dive into 1–2 components.
  6. Bottlenecks & tradeoffs — what breaks at 10×, how to scale and recover.
Exercise (do it on paper, then check yourself): Estimate QPS (avg + peak), storage/year, and read bandwidth for a photo-sharing service with 50M DAU, each uploading 1 photo/day (avg 1.5 MB) and viewing 50 photos/day.
