Week 2 — Networking, Load Balancing & APIs

How a request actually travels, how you spread traffic across machines, and how services talk to each other. By end of week: sketch the full path browser → DNS → LB → gateway → server, pick L4 vs L7 with a reason, and design a clean, secured REST API.

1. Life of a request

  1. DNS lookup: browser resolves example.com → IP. Cached at browser, OS, resolver, and authoritative levels (deep dive in §2).
  2. TCP handshake (SYN / SYN-ACK / ACK), then TLS handshake for HTTPS — TLS 1.3 (today's standard) costs one round trip, and 0-RTT resumption makes repeat connections nearly free; still, connection reuse (keep-alive) matters.
  3. HTTP request hits an edge (CDN / reverse proxy / load balancer), which routes to an app server.
  4. Response flows back; the browser renders.
  • HTTP/1.1: one request at a time per connection (head-of-line blocking) → browsers open ~6 connections.
  • HTTP/2: multiplexes many streams over one TCP connection; header compression.
  • HTTP/3 (QUIC): runs on UDP; removes TCP head-of-line blocking; combines transport + TLS handshake. No longer exotic — roughly a third of web traffic, on by default at major CDNs and browsers.

2. DNS — the internet's phone book

  • Resolution path (on a full miss): browser cache → OS cache → recursive resolver (your ISP's, or 8.8.8.8) → root server → TLD server (.com) → authoritative server for the domain. Every level caches per the record's TTL — DNS is a giant distributed, eventually-consistent database.
  • Records to know: A/AAAA (name → IPv4/IPv6), CNAME (alias to another name), NS (who's authoritative), TXT (domain verification, SPF).
  • TTL tradeoff: long TTL = fewer lookups but slow to change; drop it to ~60 s before a planned migration. Clients don't always honor TTLs, so DNS failover takes minutes, not seconds — which is why real failover lives at the LB layer, not in DNS.
  • DNS as a load balancer: multiple A records (round-robin), weighted answers, or GeoDNS — answer with the nearest region's IP. Coarse, but it's the only balancing that acts before traffic reaches you; every multi-region design starts here.

3. Proxies

  • Forward proxy: sits in front of clients — hides the client from the server (corporate egress, filtering, anonymity).
  • Reverse proxy: sits in front of servers — hides the fleet from clients. Does TLS termination, compression, caching, rate limiting, routing (nginx, Envoy, HAProxy).
  • A load balancer is a reverse proxy whose main job is distribution; most real deployments combine the roles.

  • Algorithms: round-robin (default), weighted round-robin (uneven servers), least connections (uneven request cost), IP/URL hash (stickiness or cache locality).
  • Health checks pull dead servers out of rotation — this is what turns redundancy into availability.
  • LBs exist at every tier: DNS → edge LB → service-to-service LB. The LB itself must not be a single point of failure (active-passive pair, or anycast).
  • Sticky sessions pin a user to one server — avoid if you can; keep servers stateless and put session state in Redis instead.

5. API gateway

  • The single front door for a microservice backend: clients hit one endpoint; the gateway routes /users/* → user service, /videos/* → video service.
  • Central place for cross-cutting concerns: authentication (validate the token once, not in 30 services), rate limiting, TLS termination, request logging, response caching, protocol translation (HTTP outside → gRPC inside).
  • Gateway vs load balancer: an LB spreads identical traffic across copies of one service; a gateway routes and applies policy across many different services. In practice they stack: edge LB → API gateway → per-service LBs. (Kong, AWS API Gateway, Envoy.)
  • Keep it thin — routing and policy only. Business logic in a gateway becomes a chokepoint everyone fears deploying. The gateway is stateless, so scale it horizontally like any web tier.

6. REST API design

  • Resources are nouns, methods are verbs: GET /users/123/orders, not GET /getUserOrders?id=123.
  • Idempotency: GET, PUT, DELETE are idempotent (safe to retry); POST is not — for payments/orders, accept an idempotency key header so retries don't double-charge.
  • Pagination: offset (?page=3) is simple but slow at depth and skips/dupes under writes; cursor-based (?after=eyJpZCI6...) is stable and O(1) — use it for feeds.
  • Versioning: /v1/... in the path is the pragmatic default.
  • Status codes: 200 OK, 201 Created, 400 bad input, 401 unauthenticated, 403 unauthorized, 404 not found, 409 conflict, 429 rate-limited, 500 server bug, 503 overloaded.

8. Server push: polling vs long-polling vs WebSockets vs SSE

How does the server tell the client something happened? (Shows up in chat, notifications, live dashboards.)

TechniqueHowTradeoff
Short pollingClient asks every N secondsSimple; wasteful; up to N sec latency
Long pollingServer holds the request open until data arrives, then client re-requestsNear-real-time without special protocols; one held connection per client
WebSocketsPersistent, full-duplex TCP connectionTrue two-way real-time (chat, games); stateful — harder to load-balance and scale
SSEServer-to-client one-way stream over HTTPSimpler than WS when the client never pushes (feeds, tickers); auto-reconnect built in
Interview default: chat / collaborative editing → WebSockets. Notifications / live scores → SSE or long polling. "Check occasionally" → polling. Say the tradeoff out loud: persistent connections cost memory per client and make servers stateful.

9. Security essentials

  • Authentication = who are you (fail → 401); authorization = what may you do (fail → 403). Say both words in every API design.
  • Sessions vs JWTs: session = random ID in a cookie, state in Redis — easy to revoke, but every request hits the session store. JWT = signed, self-contained token — stateless, any server verifies it locally, but revocation is hard: keep expiry short (~15 min) and pair with a refresh token.
  • OAuth 2.0 / OIDC: delegated auth — "Sign in with Google" without ever seeing the password. Know the shape: redirect to provider → user consents → app receives an authorization code → exchanges it for tokens.
  • API keys identify applications (service-to-service calls, public-API quotas); user auth identifies people.
  • Encrypt everywhere: TLS in transit, encryption at rest, and passwords hashed with bcrypt/argon2 — never encrypted, never plain.
  • Passkeys (WebAuthn) are the modern login default — phishing-resistant public-key credentials synced by the platform (Google/Apple accounts are passkey-first now). Mentioning "passkeys with password fallback" beats "passwords + SMS 2FA" in a 2026 interview.
  • Edge defenses: WAF for injection/bot patterns, per-user/IP rate limiting (week 7 builds one), DDoS absorption at the CDN. In interviews one sentence usually covers it: "auth at the gateway via JWT, TLS everywhere, rate limiting at the edge."
Exercise: Design the REST API for a pastebin: create paste (with optional expiry + custom alias), fetch paste, delete paste. Specify endpoints, methods, status codes, and how you'd paginate a user's paste list. Then decide: where does a load balancer go, L4 or L7, which algorithm — and where do requests get authenticated?
