Week 8 — Case Studies: News Feed & Instagram

The most-asked interview design. The entire problem reduces to one decision — fan-out on write vs fan-out on read — and how you handle the celebrity who breaks it.

Case 1 — Twitter (X) / news feed

Requirements & estimates

  • Functional: post tweet; follow users; home timeline = recent tweets from everyone you follow, newest-ish first.
  • Non-functional: feed load < 200 ms; extreme read skew (browse ≫ post); eventual consistency is fine — nobody notices a tweet arriving 5 s late. Availability over consistency.
  • Estimate: 200M DAU, 2 posts/day → ~5k writes/sec; each user loads the feed 10×/day → ~25k feed reads/sec, each needing ~dozens of tweets. Naive per-load computation = death by fan-in.

The core decision — fan-out on write vs read

Fan-out on write (push)Fan-out on read (pull)
On tweetInsert tweet ID into every follower's precomputed feed list (Redis)Just store the tweet
On feed loadRead your list — one cache read, blazing fastFetch recent tweets from all N followees, merge, sort — N reads per load
Cost lands onWrite time (async, invisible to users)Read time (the thing we promised would be fast)
Breaks whenCelebrity posts: 100M followers = 100M list inserts for one tweetUser follows thousands → slow merges every load
The answer interviewers want — hybrid: fan-out on write for normal users (99.9%+), but don't fan out celebrities. Mark accounts over a follower threshold; at read time, merge your precomputed list with a pull of the handful of celebrities you follow (their recent tweets are guaranteed cache-hot anyway). Inactive users (no login in 30 days) get no push either — compute on return.

Design

  • Post path: write tweet to DB (sharded by tweet_id, Snowflake IDs so ID order ≈ time order) → publish to queue → fan-out workers push the ID into followers' Redis lists (capped at ~800 entries; deep history falls back to pull).
  • Read path: Redis feed:{user_id} → list of tweet IDs → hydrate from a tweet-content cache → merge celebrity pull → return. Cursor pagination by tweet ID.
  • Social graph: follows(follower_id, followee_id) sharded by follower for "who do I follow", plus an inverted copy sharded by followee for fan-out. (Two copies — writes update both.)
  • Ranking: start chronological; mention that ranked feeds score candidates with a model at read time — don't go deeper unless asked.

Case 2 — Instagram (photo feed)

Same skeleton as Twitter plus a media pipeline — reuse week 6's design:

  • Upload: original → object storage (S3); queue → workers make thumbnail/medium/full sizes; DB row holds metadata + S3 keys only (never blobs in the DB).
  • Serve every image via CDN; feed responses carry CDN URLs.
  • Feed = fan-out on write of photo IDs (not images) exactly as above; celebrity exception applies doubly (Instagram is celebrity-dense).
  • Estimate flavor: 100M uploads/day × 2 MB avg ≈ 200 TB/day of media — storage cost and CDN egress dominate this system, not the metadata DB.

Likes & counters (classic follow-up)

  • Naive UPDATE count = count + 1 on a viral post = one hot row taking thousands of writes/sec.
  • Fix: shard the counter (N sub-counters, read sums them) or buffer increments in Redis and flush batches to the DB. Counts can be seconds-stale — say so.
Exercise (45 min, then self-critique): Design Twitter end-to-end with the hybrid fan-out. Then stress it: what breaks at 10× DAU? Where does replication lag show up visibly (hint: "I posted but my own profile doesn't show it")? How do you fix read-your-own-writes for the author?

Self-check

  • Fan-out on write vs read — cost of each, and who pays it?
  • Why exactly does push break for celebrities, and what's the hybrid?
  • Why Snowflake IDs instead of auto-increment for tweets?
  • Why cap the precomputed feed list at ~800 entries?
  • Where do the actual image bytes live, and what's in the DB?
  • How do you count likes on a viral post without melting a row?
