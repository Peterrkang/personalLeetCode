Week 5 — Partitioning, Sharding & Consistent Hashing

What you do when the data no longer fits — or the writes no longer fit — on one machine. By end of week: pick a shard key and defend it, draw the consistent-hashing ring cold, and answer "what about the celebrity?"

1. Why and when to shard

  • Shard when a single node hits a wall: data size (10 TB table), write throughput (replicas only scale reads), or working set > RAM.
  • Sharding = horizontal partitioning: split rows across machines, each shard holds a subset. (Vertical partitioning = split columns/tables by feature — do that first, it's cheaper.)
  • Costs you pay forever after: cross-shard queries and joins (scatter-gather), cross-shard transactions (2PC pain — avoid), rebalancing, and operational complexity. Shard as late as possible.

2. Partitioning strategies

StrategyHowGoodBad
RangeKeys A–M → shard 1, N–Z → shard 2 (or by time)Range scans stay on one shardHotspots: new/time-ordered keys all hit the last shard
Hashhash(key) % N → shardEven spreadRange queries hit every shard; naive mod breaks on resize (see §3)
DirectoryLookup service maps key → shardTotal flexibility; migrate keys individuallyThe directory is an extra hop and a single point of failure
  • Choosing the shard key is the whole game. You want: (1) even data + traffic distribution, (2) the common query answerable on one shard.
  • Example: shard user data by user_id → "load this user's stuff" is one shard. Shard tweets by tweet_id → even spread, but "this user's timeline" is scatter-gather. Every choice trades something; say the tradeoff.

3. Consistent hashing — know it cold

Problem: with hash(key) % N, changing N (adding/removing a node) remaps almost every key → a full-cluster data shuffle or total cache miss storm.

  1. Picture a ring of hash values 0 → 2³² wrapping around.
  2. Hash each node onto the ring. Hash each key onto the ring.
  3. A key belongs to the first node clockwise from it.
  4. Add a node → it takes over only the arc between it and its predecessor: ~1/N of keys move, everything else stays put. Remove a node → its keys slide to the next node clockwise.
  • Virtual nodes: hash each physical node to ~100–200 points on the ring. Fixes lumpy distribution, lets a big machine own more vnodes, and spreads a dead node's load across everyone instead of dumping it on one neighbor.
  • Used by Cassandra, DynamoDB, Memcached clients, most distributed caches/routers.
  • Replication on the ring: store each key on its node plus the next R−1 nodes clockwise.

4. Hotspots & the celebrity problem

  • Even a perfect hash spreads keys, not load — one viral user/post can melt its shard.
  • Fixes, in escalating order:
    1. Cache the hot entity in front of the shard (often enough — hot data is by definition cacheable).
    2. Replicate the hot key to multiple nodes and load-balance reads across them.
    3. Split the key: append a random suffix (celeb123#0..9) so writes spread over 10 sub-keys; readers aggregate. (For write-hot keys, e.g. like-counters.)
    4. Isolate: give the whale its own dedicated shard (real systems literally special-case celebrities).
  • Rebalancing: plan for it up front. Trick: create many more logical partitions than nodes (e.g. 1024 partitions on 8 nodes) and move whole partitions when nodes join — no key-level rehashing ever.

Worked example — sharding a 10 TB users table

  • 10 TB across shards targeting ≤ 1 TB each (headroom for growth + working set in RAM) → 16 logical shards, mapped onto 8 physical nodes via consistent hashing with vnodes.
  • Shard key: user_id (hash) — profile reads, settings, and auth all hit one shard.
  • Query "find user by email" now can't route → keep a small global index table email → user_id (itself sharded by email hash).
  • A user goes viral: their profile page reads 100k QPS → the profile is served from Redis (fix 1), and their follower list reads are replicated across 3 nodes (fix 2). Writes (new followers) go through a queue to absorb the spike.
