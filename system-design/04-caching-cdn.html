Week 4 — Caching, CDN & Replication

The cheapest 10–100× win in any read-heavy system. By end of week: place caches at every layer of a design, pick a write strategy, and explain leader-follower replication with its one big gotcha.

1. Where caches live

Client (browser) → CDN → reverse proxy → application cache (Redis/Memcached) → DB's own buffer pool. In interviews, "add a cache" usually means the application layer: a Redis lookup at ~0.5 ms vs a DB query at ~10–50 ms.

  • Caching works because of skew: the 80/20 rule — a small fraction of hot keys serves most reads. Measure with hit ratio; a cache below ~80% hits usually needs rethinking (keys too unique, TTL too short).

2. Read & write strategies

StrategyHow it worksTradeoff
Cache-aside (lazy)App reads cache; on miss, reads DB and fills cacheThe default. Only hot data cached; first read is slow; app owns the logic
Read-throughCache itself loads from DB on missSimpler app code; cache library must know the DB
Write-throughWrite cache + DB synchronouslyCache always fresh; writes pay double latency
Write-backWrite cache only; flush to DB asyncFastest writes, absorbs bursts; can lose data if cache dies before flush
Write-aroundWrite DB only; cache fills on readKeeps write-once data from polluting the cache

3. Eviction & invalidation

  • Eviction (cache is full): LRU is the default — drop the least-recently-used key. LFU (least frequently used) resists one-off scans; FIFO is rarely right.
  • Invalidation (data changed): the famously hard problem. Options: TTL (simple; bounded staleness), delete-on-write (app deletes the key when it updates the DB — preferred over updating the cache, which races), or event-driven purge.
  • Classic failure modes worth naming in interviews:
    • Thundering herd / cache stampede: a hot key expires and 10k requests hit the DB at once. Fix: per-key lock ("only one request recomputes"), or staggered/soft TTLs.
    • Cache penetration: requests for keys that don't exist always miss to the DB. Fix: cache the negative result, or a Bloom filter in front (§4).
    • Hot key: one celebrity key overloads a single cache node. Fix: replicate the key across nodes or add a local in-process cache.

4. Bloom filters

  • A probabilistic set in tiny memory: a bit array + k hash functions. Add a key → set its k bits. Query → check the k bits: any bit 0 = definitely not present; all 1 = probably present.
  • False positives only, never false negatives — that asymmetry is the whole point. ~10 bits per element gives a ~1% false-positive rate: a billion keys fit in ~1.2 GB of RAM. Standard filters can't delete (bits are shared between keys); a counting variant can.
  • Where it earns its keep: cache penetration (the fix from §3 — skip the DB for keys that can't exist), LSM databases (Cassandra/RocksDB consult a filter before touching each SSTable on disk), "username taken?" pre-checks, Chrome's malicious-URL list.
  • Interview line: "a Bloom filter in front — if it says no, the key definitely isn't there and we skip the lookup; the ~1% false positives just do the lookup they would have done anyway."

5. CDN

  • A CDN is a globally distributed cache for content, terminating requests near the user — turns a 150 ms cross-ocean round trip into ~10 ms to an edge PoP.
  • Pull CDN (default): edge fetches from your origin on first miss, then caches per Cache-Control. Easy; first request per region is slow.
  • Push CDN: you upload content to the CDN ahead of time. For large, predictable assets (video releases).
  • Serve static assets (images, JS, video segments) from CDN always; cache-bust with hashed filenames (app.3f2a1.js + immutable TTL). Dynamic API responses can also be edge-cached briefly (e.g. 5 s) to absorb read storms.

6. Redundancy & replication

  • Redundancy = extra copies of components (no single point of failure). Replication = keeping the copies of data in sync.
  • Leader-follower (primary-replica): writes go to the leader; followers replicate the log and serve reads. Read scaling for free; on leader death, promote a follower (failover).
  • Replication is usually asynchronous → replication lag. The classic bug: user writes a comment (leader), refreshes, reads from a lagging replica — comment "disappears." Fix: read-your-own-writes — pin a user's reads to the leader briefly after they write.
  • Multi-leader: writes accepted in several regions — lower write latency worldwide, but concurrent writes to the same row conflict and need resolution (last-write-wins, CRDTs). Avoid unless you need it.
  • Followers double as failover targets and backups — replication is both a scaling and an availability tool.

Worked example — bolting caching onto the e-commerce design

  • Product pages: cache-aside Redis, key product:{id}, TTL 5 min + delete-on-write when a merchant edits. Read QPS on the DB drops ~95%.
  • Product images: CDN with hashed URLs, 1-year TTL.
  • Stock counts: don't cache (or cache seconds only) — overselling is worse than a slow read. Correctness beats speed here.
  • DB: 1 leader + 2 read replicas; browsing reads hit replicas, checkout reads/writes hit the leader.
Exercise: Your product cache shows stale prices for up to 5 minutes after a merchant update. Walk through three invalidation designs (shorter TTL, delete-on-write, event-driven purge) and their failure modes. Then: what happens when the "daily deals" key expires at noon under 50k QPS, and how do you prevent it?
