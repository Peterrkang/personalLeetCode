Week 7 — Case Studies: URL Shortener & Rate Limiter

The two canonical warm-up designs. Do each end-to-end with the 6-step framework, 45 minutes on paper. These recycle everything from weeks 1–6.

Case 1 — URL shortener (TinyURL)

Requirements & estimates

  • Functional: shorten URL (optional custom alias, optional expiry) → short link; redirect; basic click analytics.
  • Non-functional: redirect must be fast (<50 ms) and highly available — a broken redirect is a dead link on someone else's site. Heavily read-skewed: assume 100:1 read:write.
  • Estimate: 100M new URLs/month ≈ 40 writes/sec; 10B redirects/month ≈ 4k reads/sec (peak ~10k). Storage: 100M × ~500 B/month ≈ 50 GB/year — tiny. This system is about read latency, not storage.

The core question: generating the short key

  • Key alphabet: base62 ([a-z A-Z 0-9]). 62⁷ ≈ 3.5 trillion — 7 chars is plenty forever.
  • Option A — hash the URL (MD5, take first 7 chars): same URL → same key (nice dedupe), but collisions need probing (check DB, re-hash with salt) — racy under concurrency.
  • Option B — counter + base62-encode: a global auto-increment ID encoded in base62. No collisions ever. Keys are guessable/sequential (usually fine; can XOR-scramble). Single counter = bottleneck → Option C.
  • Option C — Key Generation Service (the Design Gurus classic): pre-generate random keys into a pool; app servers grab batches (e.g. 10k keys) so there's no per-request coordination and no collision check at write time. Mark batch used; if a server dies, its unused batch is lost — who cares, the space is huge.
  • Custom alias: just an insert with a uniqueness constraint; 409 on conflict.

Design

  • POST /urls {long_url, alias?, expiry?} → {short}; GET /{short} → 301/302 redirect. (301 = permanent, browsers cache it and you lose analytics; 302 keeps every click visible. Say this tradeoff.)
  • Storage: key-value shaped — short → {long_url, owner, expiry, created}. Any KV store / DynamoDB / sharded SQL works; shard by hash of short key.
  • Cache the redirects: cache-aside Redis on short → long_url; the read skew means ~all redirects are cache hits. CDN/edge caching can take it further.
  • Analytics: don't write on the redirect path — publish a click event to a queue/Kafka; aggregate async.
  • Expiry: lazy delete on read + background sweep (active per-key timers don't scale).

Case 2 — Rate limiter

Requirements

  • Limit requests per client (per user / API key / IP): e.g. 100 req/min. Return 429 with a Retry-After header. Must add ~no latency (<1–2 ms), work across a distributed fleet, and fail sensibly.
  • Where: in the API gateway / middleware, before expensive work.

Algorithms (know all four, recommend one)

AlgorithmHowVerdict
Fixed windowCounter per minute-bucketSimple, but burst at the boundary: 100 at 0:59 + 100 at 1:01 = 200 in 2 s
Sliding window logStore every request timestamp, count last 60 sExact but O(rate) memory per client — expensive
Sliding window counterWeighted blend of current + previous fixed windowsGood approximation, tiny memory — common production choice
Token bucketBucket holds B tokens, refills at R/sec; request takes a token or is rejectedThe default answer — allows controlled bursts (B) with sustained rate (R); O(1) memory: just {tokens, last_refill}

(Leaky bucket = token bucket's cousin: requests drain from a queue at a fixed rate — smooths output perfectly but adds queueing delay.)

Distributed rate limiting

  • Counters must be shared across API servers → Redis: INCR + EXPIRE, or token bucket as an atomic Lua script (read-modify-write must be atomic or concurrent requests double-spend).
  • Latency trick: local in-memory pre-check + async sync to Redis — slightly loose limits, much faster.
  • Fail open or closed? If Redis is down: fail open (let traffic through) for user-facing APIs — availability over strictness; fail closed for abuse-critical endpoints (login attempts). Name this decision unprompted.
  • Return headers: X-RateLimit-Limit, X-RateLimit-Remaining, Retry-After.
Exercise (do both, timed 45 min each): (1) Full TinyURL on paper — include the 301 vs 302 decision, KGS, cache, and what breaks at 10× read traffic. (2) Rate limiter for a public API with free (100/min) and paid (10k/min) tiers — include the Redis data model, the atomicity problem, and fail-open vs fail-closed.

Self-check

  • Why base62, and why is 7 characters enough?
  • What problem does the Key Generation Service remove from the write path?
  • 301 vs 302 — what do you gain and lose with each?
  • Why does fixed-window rate limiting allow 2× bursts?
  • Token bucket: what do B and R each control?
  • Why must the Redis counter update be atomic, and what breaks if it isn't?
