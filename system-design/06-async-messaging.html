Week 6 — Async Processing & Messaging

Anything that doesn't have to happen before the user sees "OK" shouldn't. By end of week: put a queue in the right place, choose queue vs stream, and explain why exactly-once is really at-least-once + idempotency.

1. Why go async

  • Decoupling: producer and consumer don't know each other, deploy independently, fail independently.
  • Buffering: the queue absorbs spikes — producers burst at 50k/sec, consumers drain steadily at 5k/sec, nobody falls over.
  • Latency: respond to the user after the essential write; do the slow parts (emails, thumbnails, fan-out, analytics) in the background.
  • The price: eventual consistency (the work isn't done at response time), plus a new component to operate and monitor (queue depth!).

2. Message queues vs event streams

Queue (SQS, RabbitMQ)Log / stream (Kafka, Kinesis)
ModelTask list: consumer takes a message, acks it, it's goneAppend-only log: consumers keep an offset; messages retained for days
ConsumersCompeting workers share one queueMany independent consumer groups each read the whole stream
ReplayNoYes — rewind the offset
OrderingUsually best-effortStrict within a partition (partition by key, e.g. user_id)
Use forJobs: send email, resize imageEvent firehose: activity events, CDC, feeding multiple systems
  • Pub/sub is the fan-out pattern: one event, many independent subscribers (order-placed → email service, inventory, analytics). Queues do it with topics/exchanges; Kafka does it natively via consumer groups.

3. Delivery guarantees

  • At-most-once: fire and forget — can lose messages. Rarely acceptable.
  • At-least-once: the default. Consumer acks after processing; if it crashes mid-work, the message is redelivered → duplicates happen.
  • Exactly-once: effectively unachievable end-to-end across arbitrary systems. The real-world answer: at-least-once delivery + idempotent consumers = exactly-once effect. (Kafka's transactions do give exactly-once within the Kafka ecosystem — consume-process-produce loops — but the moment you touch an external system, you're back to idempotency.)
  • Idempotent consumer: processing the same message twice is harmless. Techniques: dedupe table keyed by message ID (skip if seen), upserts instead of inserts, conditional updates ("set status=paid where status=pending").

4. Operating a pipeline (the senior signals)

  • Backpressure: producers outpace consumers → queue grows without bound. Watch queue depth / consumer lag; respond by autoscaling consumers, rate-limiting producers, or shedding load.
  • Dead-letter queue (DLQ): after N failed attempts, park the message aside instead of retrying forever — one poison message must not block the pipeline. Alert on DLQ size, inspect, replay after the bug fix.
  • Retries with exponential backoff + jitter — immediate tight retries amplify outages.
  • Ordering: if user events must be processed in order, partition by user_id so one consumer sees that user's events sequentially. Global ordering doesn't scale; per-key ordering does.
  • Batch vs stream processing: batch = periodic jobs over bounded data (nightly reports); stream = continuous processing of unbounded data with seconds latency (live dashboards, fraud detection).

Worked example — async image-processing pipeline

  1. Client uploads image → API stores original in object storage (S3), writes a DB row status=processing, publishes {image_id} to the queue, returns 202 Accepted immediately.
  2. Worker fleet consumes: download original → generate 3 sizes → upload to S3 → update row status=ready → publish image.ready for anyone who cares (notifications, feed).
  3. Duplicates (at-least-once): resizing is naturally idempotent — same input, same output paths, upsert the row. Safe.
  4. Poison message (corrupt image crashes the worker): 3 retries → DLQ → alert. Pipeline keeps flowing.
  5. Black Friday spike: uploads 20× — queue depth grows, autoscaler adds workers, users see "processing…" a little longer. Nothing falls over. That's the whole point.
Exercise: Design "order placed" processing for e-commerce: payment capture, inventory decrement, confirmation email, analytics. Which steps must be synchronous? Which go through the queue? Where exactly do duplicates bite, and what makes each consumer idempotent?
