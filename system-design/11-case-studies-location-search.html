Week 11 — Case Studies: Uber & Typeahead

Two specialized-index designs: geospatial ("what's near me?") and prefix search ("what starts with…?"). The insight in both is the same — a plain B-tree can't answer the question, so you build the index shape that can.

Case 1 — Uber / Lyft / Yelp (proximity)

Why not just a database?

WHERE lat BETWEEN … AND lng BETWEEN … can't use one B-tree efficiently — each index narrows one dimension. You need a spatial index that maps 2D → something indexable.

IndexIdeaNotes
GeohashInterleave lat/lng bits → base32 string; longer = smaller cell. Nearby points usually share prefixesString prefix search = cheap (works in Redis). Gotcha: neighbors across cell borders don't share a prefix → always query the cell + 8 neighbors
QuadtreeRecursively split the map into 4 quadrants until each leaf holds ≤ K pointsAdapts to density: Manhattan = tiny cells, Nevada = huge. Yelp-style favorite
S2 / H3Production cell libraries (Google/Uber)Name-drop; don't derive

Design

  • Drivers ping location every ~4 s → millions of writes/sec that are instantly stale → keep the live index in memory (Redis: geohash cell → set of driver IDs), not in a durable DB. Persist only trip events. It's fine to lose the live index — it rebuilds in seconds from pings.
  • Matching: rider requests → compute rider's cell → fetch drivers in cell + neighbors → rank by ETA (road distance, not straight-line) → offer to driver(s); lock the driver during the offer (no double-dispatch).
  • Scale by geography: shard the index by region/city — queries are inherently local. A hot city (concert lets out) stays contained.
  • Surge / supply-demand: per-cell counters of open requests vs available drivers — another thing cells give you for free.
  • Yelp variant: places don't move → static quadtree, heavily cached, rebuilt on business updates; add ratings/filters at rank time.

Case 2 — Typeahead / autocomplete

Requirements

  • Suggestions within ~100 ms per keystroke; top 5–10 by popularity; freshness can lag (yesterday's trends are fine; call this out).
  • Scale: every keystroke of every search user hits this — read QPS is enormous; it must be cheap per query.

Design — trie + precomputed top-K

  • A trie over popular queries. The trick: store the top-K completions at every node, precomputed — a lookup is walk-the-prefix + return the stored list. O(prefix length), zero ranking work at query time.
  • Don't update the trie online. Log queries → aggregate counts offline (batch/stream job) → rebuild the trie periodically (hourly/daily) → hot-swap. Serving stays read-only and lock-free.
  • Sharding the trie: by prefix range (a–i / j–r / s–z, tuned by traffic so 's' doesn't melt one shard). Each shard replicated for read throughput.
  • Cache the first 1–3 characters (there are only ~26–26³ of them — they cover a huge share of traffic; even a CDN can serve these).
  • Client hygiene: debounce keystrokes, cancel in-flight requests, cache results locally.
  • Extras when asked: per-user personalization (blend a small user-history trie), trending injection (fast lane for sudden spikes), filtering (never suggest blocked terms).
Shared lesson: both designs precompute the answer shape. Uber buckets the world into cells so "near me" is a set lookup; typeahead stores top-K at every node so ranking already happened. When latency budgets are tight, move work from read time to write/build time.
Exercise (45 min): Design typeahead end-to-end: trie with top-K at nodes, offline rebuild pipeline, prefix sharding, first-char caching. Deep-dive: a breaking-news term needs to surface within minutes — what's your fast path around the daily rebuild? Then sketch Uber's driver index and explain why it lives in RAM.

Self-check

  • Why can't a composite B-tree index answer "nearest 10 drivers"?
  • Geohash vs quadtree — one strength of each, and the geohash border gotcha?
  • Why is it acceptable — desirable, even — that Uber's live index is volatile?
  • What exactly is stored at each trie node, and why does that make queries O(L)?
  • Why rebuild the trie offline instead of updating counts per query?
  • How do both systems trade write/build-time work for read-time speed?
