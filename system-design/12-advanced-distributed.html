Week 12 — Advanced Theory & Senior Depth

CAP applied (not recited), consistency models, quorums, the distributed-systems toolbox, and the resilience patterns that separate senior answers from mid-level ones.

1. CAP — applied, not memorized

  • During a network partition, choose: refuse some requests (CP — stay consistent) or answer with possibly-stale data (AP — stay available). No partition → no forced choice.
  • It's per-operation, not per-company: your payment ledger is CP; your product-view counter is AP. Saying this is the senior move.
  • PACELC finishes the thought: if Partition → A vs C; Else → Latency vs Consistency. Even with no failures, strong consistency costs coordination round trips. DynamoDB default = PA/EL; a single-leader SQL setup ≈ PC/EC.

2. Consistency models (strongest → weakest)

ModelGuaranteeExample need
Linearizable (strong)Every read sees the latest acknowledged write, everywhereAccount balances, inventory at checkout, locks
CausalCausally related events appear in order (reply never precedes its post)Comment threads, chat
Read-your-own-writesYou see your own updates; others may lagProfile edits, posting (the replication-lag fix from week 4)
EventualStop writing and replicas converge… eventuallyLike counts, view counts, DNS

Weaker = faster + more available. Pick per data type and justify — "eventual everywhere" and "strong everywhere" are both wrong answers.

3. Quorums

  • N replicas; a write must be acked by W, a read consults R. If R + W > N, read and write sets overlap → reads see the newest write.
  • N=3: W=2, R=2 is the balanced classic. W=1 = fast risky writes; W=3 = durable slow writes. Tune per workload — this is Cassandra/Dynamo consistency-level tuning in one line.

4. Consensus & coordination (conceptual level)

  • The problem: machines that crash and a network that drops messages must still agree on one value (who's leader, what's committed).
  • Raft (know its shape): nodes are follower / candidate / leader; leader elected by majority vote with randomized election timeouts; all writes flow through the leader, committed once a majority replicates them. Majority = any two quorums intersect → no split-brain (two leaders during a partition — the minority side can't commit).
  • Failure detection is heartbeats: leader pings followers; silence past a timeout → new election. (Same primitive as LB health checks and chat presence — it's heartbeats all the way down.)
  • You don't implement consensus; you rent it: ZooKeeper / etcd for leader election, distributed locks, service discovery, config. Use it for coordination metadata, never for data-path traffic. (The 2026 trend is embedding Raft instead of running a separate coordinator — Kafka dropped ZooKeeper entirely for built-in KRaft as of Kafka 4.0; CockroachDB and etcd itself are Raft-native.)

5. Heartbeats & checksums — detecting failure and corruption

  • Heartbeat: a periodic "I'm alive" signal — the only way a distributed system learns a node died. Push (node reports to a monitor) or pull (monitor probes an endpoint); miss ~3 consecutive intervals → declare the node dead and act (re-route, re-replicate, re-elect).
  • The tuning tradeoff: short interval/timeout = fast detection but false positives — a GC pause or network blip gets a healthy node declared dead and its data pointlessly re-replicated. Long = slow failover. A timeout can never distinguish dead from slow; that ambiguity is where split-brain comes from, and why section 4's majority vote exists.
  • Same primitive everywhere: LB health checks (week 2), Raft leader liveness (above), GFS/HDFS chunkserver liveness (week 10), chat presence (week 9).
  • Checksum: a hash stored alongside each block of data and verified on every read. Disks silently rot, RAM flips bits, and TCP's 16-bit checksum misses errors at scale — so verify end-to-end: compute where data is written, check where it's read. Mismatch → discard that replica, read another copy, re-replicate.
  • S3 and HDFS checksum every block and also run background scrubbing — re-verifying cold data so all three replicas don't rot unnoticed.

6. Small tools worth name-dropping

  • Bloom filter: probabilistic set — "definitely not present" or "probably present" in tiny memory. Skips useless disk lookups (LSM stores like Cassandra), pre-filters cache penetration, "username taken?" pre-check. (Full treatment in week 4.)
  • Gossip protocol: nodes exchange state with random peers; cluster membership/failure info spreads in O(log n) rounds without a coordinator (Cassandra, Dynamo).
  • Distributed tracing: a request ID propagated through every hop (OpenTelemetry) — how you debug a 30-service latency mystery.

7. Resilience patterns (say these unprompted)

  • Timeouts everywhere — an un-timed-out call is a thread leak waiting for an outage.
  • Retries with exponential backoff + jitter — and only on idempotent operations; retry storms without jitter synchronize and DDoS your own recovery.
  • Circuit breaker: after N failures to a dependency, fail fast for a cooldown (half-open probe to recover) — stops cascading failure and gives the sick service air.
  • Bulkheads: partition resources (thread pools, connections) per dependency so one slow downstream can't drown everything.
  • Graceful degradation + load shedding: feed unavailable → serve cached feed; overloaded → drop low-priority traffic (analytics) before user-facing traffic. Decide the sacrifice order in advance.
  • Observability: metrics (RED: rate, errors, duration + queue depths), structured logs, traces; alert on SLO burn, not on every blip. SLA = external promise, SLO = internal target, SLI = the measurement.
The senior close: for any design, be able to answer — What's the consistency model of each store, and why? What happens when each box dies? What do we shed first at 10× load? How would we know it's failing before users tweet about it?
Exercise: Take your Week 8 Twitter design and harden it: label every datastore CP or AP with a reason; add circuit breakers where fan-out calls dependencies; define three SLOs (feed latency, post durability, availability) with SLIs; and describe the load-shedding order during a viral event.

Self-check

  • State CAP in one sentence, then apply it to a payment ledger vs a view counter.
  • What does PACELC add that CAP misses?
  • Why does R + W > N guarantee fresh reads?
  • How does majority voting prevent split-brain?
  • Why does an aggressive heartbeat timeout cause more failures than it catches?
  • TCP already has checksums — why do S3 and HDFS still checksum every block?
  • What can a Bloom filter never tell you?
  • Circuit breaker vs retry — when does each make an outage worse?
